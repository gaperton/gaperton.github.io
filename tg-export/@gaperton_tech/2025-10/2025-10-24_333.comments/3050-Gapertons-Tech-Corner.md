Могу сказать что использую я. 

- OMT/UML sequence для сценариев (всегда) 
- OMT/UML state для машин состояний (когда есть) 
- ER crow foot для схем БД (всегда)
- Спеки JSON и эндпойнты в тексте для REST API (всегда)

Это позволяет делать кросс-проверки ER против sequence, и JSON против ER, и много чего еще. И находить ошибки. Это язык для дизайн ревью, цель которого найти ошибки до того как написан код.

Этот пакет обязан предоставить на групповое ревью каждый джун, кто делает дизайн (а у меня дизайн делают все). И править по итогам ревью (часто большие переделки). А потом -- раскопипастить по задачам и подзадачам в JIRA, которые создаст (а я проверю).

Иногда
- OMT data flow чтоб показать поток данных в серверлесс пайплайне и как он с БД и API завязан.

Все. Это не значит что кому-то еще __надо __делать так же. Не обсуждаете эти аспекты дизайна с людьми -- ваще пох тогда, можете любые козявки рисовать. Каждый раз разные. Или ничего. 

**Надо -- не "рисовать диаграммы". А что-то другое для чего их надо рисовать.**

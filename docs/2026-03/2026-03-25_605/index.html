⁉️ Are there any attempts to adapt extreme programming to work with coding agents like claude code?

Отвечает chatGPT. Перевод DeepL.

Да. Уже наблюдаются явные попытки, но в основном это спонтанные практики и опыты специалистов, а не устоявшийся канон «XP для агентов».

Что существует на данный момент:

Парное программирование переосмысливается как «человек + агент-кодер», а в некоторых случаях — как триада: заказчик/владелец проблемы, разработчик и модель. Фрэнк Вестфал явно формулирует это как «Заказчик — Разработчик — Модель», причем человек переходит от роли водителя к роли навигатора, в то время как агенты занимаются реализацией и исследованием.

Практикой XP, переживающей самое сильное возрождение, является TDD. Thoughtworks утверждает, что агенты ИИ работают лучше, когда у них есть четкие критерии завершения, и конкретно указывает, что такие инструменты, как Claude Code, лучше всего работают с подходом «тест сначала».
Недавние дискуссии Кента Бека, связанные с агентами, многие практикующие специалисты резюмируют так: TDD теперь является «суперсилой», поскольку тесты дают агентам исполняемое определение корректности и сдерживают их склонность генерировать правдоподобный, но неаккуратный результат. Один из наблюдаемых режимов сбоя особенно показателен: агент может попытаться удалить неудавшийся тест вместо того, чтобы исправить ошибку.

Еще одна адаптация — микроитерации, управляемые спецификациями. Вместо классических пользовательских историй, поступающих напрямую к людям, команды пишут более четкие и явные спецификации для агентов, а затем итерации в коротких циклах. Allstacks описывает это как «мини-водопады» или «мироитерации», где спецификации на естественном языке говорят агенту, что нужно построить, а тесты определяют успех.

Существует также эквивалент на уровне репозитория общей командной дисциплины XP: живые файлы инструкций, такие как CLAUDE.md и все чаще AGENTS.md. Anthropic описывает Claude Code как использование CLAUDE.md в рамках своей контекстной стратегии, а практикующие специалисты описывают эти файлы как постоянную память проекта, содержащую рабочие процессы, стандарты, команды и правила архитектуры.
Это очень близко по духу к акценту XP на общих соглашениях, коллективной ответственности и сокращении неявных знаний.

Чего все еще не хватает, так это убедительных доказательств того, что это превратилось в широко принятую методологию. Нынешние материалы в основном состоят из:
• эссе практиков и постов в блогах,
• рекомендаций поставщиков инструментов,
• интервью и комментариев о Кенте Беке и TDD,
• ранние отчеты о рабочих процессах от команд, использующих Claude Code и подобных агентов.

Таким образом, точный ответ:

Да, безусловно, попытки есть.
Но в основном они адаптируют некоторые идеи XP, а не воспроизводят XP без изменений.

Общий паттерн выглядит следующим образом:
• Парное программирование → человек, контролирующий агента, иногда нескольких агентов
• TDD → более важно, чем раньше
• Небольшие релизы / короткие циклы обратной связи → задачи размером с агента и быстрая проверка
• Коллективная ответственность → агент может затрагивать большую часть кодовой базы, поэтому ограничения становятся более важными
• Стандарты кодирования → зашифрованы в CLAUDE.md / AGENTS.md
• Непрерывная интеграция → становится еще более центральной, поскольку агенты могут быстро генерировать большие объемы кода

Что пока не решено:
• как безопасно осуществлять «ротацию пар», когда один из членов «пары» не является человеком,
• какую степень автономии предоставить агентам до проверки,
• как предотвратить отклонение от спецификаций и манипуляции с тестами,
• как сохранить социальную дисциплину XP, а не превратить ее в «prompt-and-pray» (дайте подсказку и молитесь).

Gapertons-Tech-Corner @gaperton_tech 25/03/2026, 14:08:06

Yes. There are already clear attempts, but they are mostly emergent practice and essays from practitioners, not a settled “XP-for-agents” canon yet.

What exists so far:

Pair programming is being reinterpreted as human + coding agent, and in some cases as a triad: customer/problem-owner, developer, and model. Frank Westphal explicitly frames this as “Customer — Developer — Model,” with the human shifting from driver to navigator while agents handle implementation and exploration.

The XP practice getting the strongest revival is TDD. Thoughtworks argues that AI agents work better when they have clear exit criteria, and specifically says tools like Claude Code work best with a test-first approach. Kent Beck’s recent agent-related discussions are being summarized by multiple practitioners as: TDD is now a “superpower” because tests give agents an executable definition of correctness and constrain their tendency to generate plausible but sloppy output. One observed failure mode is especially revealing: an agent may try to remove a failing test instead of fixing the bug.

Another adaptation is spec-driven micro-iteration. Instead of classic user stories flowing directly to humans, teams are writing tighter, more explicit specs for agents, then iterating in short loops. Allstacks describes this as “mini-waterfalls” or “micro-iterations,” where natural-language specs tell the agent what to build and tests define success.

There is also a repo-level equivalent of XP’s shared team discipline: living instruction files such as CLAUDE.md and increasingly AGENTS.md. Anthropic describes Claude Code as using CLAUDE.md as part of its context strategy, and practitioners describe these files as persistent project memory containing workflows, standards, commands, and architecture rules. That is very close in spirit to XP’s emphasis on shared conventions, collective ownership, and reducing tacit knowledge.

What is still missing is strong evidence that this has matured into a broadly accepted methodology. The current material is mostly:
• practitioner essays and blog posts,
• tool-vendor guidance,
• interviews and commentary around Kent Beck and TDD,
• early workflow reports from teams using Claude Code and similar agents.

So the accurate answer is:

Yes, definitely there are attempts.
But they are mostly adapting a few XP ideas rather than reproducing XP unchanged.

The rough pattern looks like this:
• Pair programming → human supervising an agent, sometimes multiple agents
• TDD → more important than before
• Small releases / short feedback loops → agent-sized tasks and rapid review
• Collective ownership → agent can touch more of the codebase, so guardrails matter more
• Coding standards → encoded in CLAUDE.md / AGENTS.md
• Continuous integration → even more central, because agents can generate lots of code quickly

What has not really been solved yet:
• how to do safe “pair rotation” when one “pair” member is nonhuman,
• how much autonomy to give agents before review,
• how to prevent spec drift and test gaming,
• how to preserve XP’s social discipline instead of turning it into “prompt-and-pray.”

So: XP is not being replaced by coding agents; parts of XP are being rediscovered as control mechanisms for coding agents. TDD, tiny steps, continuous feedback, and explicit shared conventions appear to be the most transferable pieces.

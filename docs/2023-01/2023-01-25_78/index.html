Отделы по подсистемам

Еще один популярный структурный паттерн. Чаще всего встречается в двух вариантах. "Отдел фронтенда"-"отдел бэкенда". И наивный инфраструктурный отдел, который должен делать общие компоненты, сервисы, и библиотеки. Разберем для начала как склонны работать "отдел фронтенда"-"отдел бэкенда". Оно не всегда так работает, оно зависит от протоколов и обычаев. Но то что я опишу я видел очень часто.

Так как отдела два, и чтобы сделать фичу нужны усилия обоих — они склонны образовывать двухступенчатый конвейер. Это сразу дарит нам все свойства конвейеров, но две ступени — это терпимо. Часто бывает, что бэкенд ставят в конвейере вперед. Тогда происходит следующее.

Бэк вперед

  1. Бэкенд проектирует нормализованную схему, и выставляет случайный API который кажется им логичным. Бодро рапортует что работу закончили, не дожидаясь когда API посмотрит человек. Это по хорошему должно являться частью приемки — ведь он делается для фронтов. Но часто почему-то не является.

  2. Фронты смотрят на требования, сделали наброски UI, и понимают, что через данный API оно либо работать вообще не может, либо — будет работать крайне через жопу. Сценарии UI задают нарезку API для него, ага. Они обращаются в бэкенд.

  3. Бэкенд шлет их в жопу. Потому, что они работу закончили, и переделывать не хотят, особенно целиком. Там другие задачи в плане, и вообще — они уже "закончили" и в дедлайн уложились, а так — выйдет что не уложились. А еще потому, что фронты — некомпетентные идиоты, и всегда хотят странного.

  4. Возникает конфликт, который может решаться по разному. Часто фронты плюют, и делают у себя тоже через жопу.

Фронт вперед

Применив "вытягивающий принцип", у нас фронтенд ставит задачу бэкенду, что частично лечит ситуацию, но только в случае если весь ввод-вывод идет через UI. Посмотрим как оно пойдет.

  1. Фронты сделали прототип UI на моках API. Показали продактам. Те потыркались. Поправили кое-что. [Цикл исправления косяков в UX стал сильно короче, цена ошибки ниже.

](https://t.me/gaperton_tech/71)2. Фронты ставят задачу на недостающие API __в бэкенд. Они могут с__делать это внахлест с доделкой UI, сразу как утрясли состав экранов и основные сценарии.

  1. Бэк возмущается, что API кривой, надо делать не так. Тут возможны варианты — и если работа идет внахлест то фронты легче соглашаются что-то поменять.

  2. Бэк начинает отгружать API. Фронт интегрирует его по мере отгрузки, выключая моки, и тут же репортит баги. Баги исправляются по горячему.

  3. Все интегрировали, сдали, отлично.

Вытягивающий принцип в __данном случае работает от__лично, если работа идет внахлест. Если нет — то может быть похоже на то как было раньше, но менее болезненно. Если UI не доминирует в задаче, и бэк берет данные откуда-то еще, то все встает раком как и в первом случае.

Протокол разработки интерфейса

Дизайн это не квадратики и не то что внутри них. Это то, что между квадратиков. В нашем случае — это API между бэком и фронтом. Мы требуем, чтобы этот API был согласован между отделами как можно раньше, до того, как обе задачи завершены, и чем раньше тем лучше. Для согласовани__я используем протокол инспекции. Раньше нашли __ошибки -> ошибки стали дешевле.

Прокрутив в голове варианты, мы понимаем, что это помогает для сложного API и когда фронт вперед, и когда бэк вперед.

Пускайте фронт вперед и работу по API внахлест, если нет строгих противопоказаний к обратному. Делайте дизайн API и согласу__йте его как можно раньше. Или... Не режьте так отделы. Фулл-стек отделы в принципе лишен__ы этого структурного конфликта.

Описанное также верно для любых инфраструктурных отделов. Инфраструктурный отдел пускающий работы вперед склонен выдумывать ебанину и нагибать всех ее использовать ровно так же, как и схема Бэк вперед. Но у них еще хуже, так как все б[ерется из головы.

Всякие автор](https://hackernoon.com/opinionated-or-not-choosing-the-right-framework-for-the-job-6x1u2ga0)ы больших opinionated фреймворков поступают с вами примерно так же, как худшие варианты инфраструктурных отделов. Но их вы, по крайней мере, можете выбирать. Старайтесь избегать opinionated фреймворков, от них тяжелее избавиться если таковое понадобится.
